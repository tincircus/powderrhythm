  <action>
    Create ticketing/index.js. This file is the process entrypoint: it requires dotenv, sets up Express, runs migrations, then starts listening.

    File structure (implement in this order):
    1. 'use strict' at top
    2. require('dotenv').config() — must be first require so env vars are available to knex.js and other modules on load
    3. Require express, path, and db from ./src/db/knex
    4. const app = express() and const PORT = process.env.PORT || 3000
    5. app.set('view engine', 'ejs') and app.set('views', path.join(__dirname, 'src/views'))
    6. app.use(express.json()) and app.use(express.urlencoded({ extended: false }))
    7. GET /health route: async handler calls await db.raw('select 1'), on success returns res.json({ status: 'ok', db: 'connected' }), on error returns res.status(503).json({ status: 'error', db: 'disconnected' })
    8. Global error handler: 4-argument function (err, req, res, next) — logs err.stack to console.error, returns res.status(500).json({ error: 'Internal server error' }). The 4-argument signature is required by Express 5 to identify this as an error handler.
    9. async function start(): awaits db.migrate.latest(), logs 'Migrations complete', then calls app.listen(PORT, callback that logs "Listening on port ${PORT}"). Wraps in try/catch: on error, console.error('Startup failed:', err) and process.exit(1).
    10. start() call at the bottom.

    CRITICAL ordering: app.listen MUST be inside the start() function AFTER await db.migrate.latest(). Never call app.listen at the module top level or before migrations resolve.

    Do NOT register any stub route files in index.js at this point — only the /health route is active in Phase 1. Stub routes will be registered in Phase 2 as they gain implementations.
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| external HTTP → GET /health | Public endpoint — read-only DB ping; no input accepted, no user data returned |
| process.env.PORT | Trusted (set by Railway or developer); controls bind address |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | GET /health error branch exposes db: 'disconnected' | accept | No credentials or PII in the error response; only confirms DB unreachable state. Acceptable for an infrastructure health check. |
| T-02-02 | Denial of Service | GET /health — rapid polling could create DB connection churn | accept | Phase 1 has no rate limiting (deferred to Phase 6 per STATE.md). Single db.raw('select 1') is cheap. Railway health checks are infrequent (every 10-30s). |
| T-02-03 | Information Disclosure | Stack trace in global error handler (console.error(err.stack)) | accept | Stack trace goes to server logs (Railway log drain), not to the HTTP response body. Response body is generic "Internal server error". No leak to client. |
| T-02-04 | Tampering | index.js registers stub routes before they have auth middleware | mitigate | Stub routes are NOT registered in index.js in Phase 1 — only /health is active. Auth-protected routes (scan, admin) are registered in Phase 4 and 5 when their requireAuth middleware is implemented. Verified by: index.js does not contain require('./src/routes') at end of Phase 1. |
</threat_model>
