<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer workstation → git | .env and dev.sqlite must never cross this boundary into version control |
| ticketing/ directory → repo root | dev.sqlite must land inside ticketing/, not the repo root — path resolution via __dirname enforces this |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | ticketing/.env (DATABASE_URL, ADMIN_PASSWORD) | mitigate | .gitignore contains .env on its own line; .env.example is the only committed credential template; verified in acceptance criteria |
| T-01-02 | Information Disclosure | ticketing/dev.sqlite (local DB file with seed data) | mitigate | .gitignore contains dev.sqlite; dev.sqlite must be listed before first git add; verified in acceptance criteria |
| T-01-03 | Information Disclosure | dev.sqlite path resolution — file landing in repo root instead of ticketing/ | mitigate | SQLite filename in knex.js uses path.join(__dirname, '..', '..', 'dev.sqlite') where __dirname = ticketing/src/db/ — resolves to ticketing/dev.sqlite regardless of process.cwd(); acceptance criteria verifies __dirname presence in source |
| T-01-04 | Elevation of Privilege | Knex instantiated multiple times creating separate connection pools | accept | Phase 1 has no routes; the single-instance rule is enforced by design (one module.exports = db). Multiple-instance risk surfaces in Phase 2+ when routes are added — mitigated then by importing from src/db/knex.js. |
</threat_model>
