<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client IP → scanRateLimiter | Per-IP rate limit uses X-Forwarded-For behind Railway's proxy; rate limiter trusts the first hop |
| Square Dashboard → POST /webhooks/square | Webhook delivery from Square is verified by HMAC (SEC-01, already implemented); rate limiter is NOT applied to webhook route |
| Railway env → process.env | Production secrets injected at container startup; exposure risk is Railway account access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Information Disclosure | UUID enumeration via POST /api/scan | mitigate | scanRateLimiter at 60 req/min/IP prevents automated scanning of the UUID space (SEC-04 / D-03 / D-04) |
| T-06-06 | Denial of Service | Rate limiter throttles Square webhook retries | accept | Limiter is route-scoped to POST /api/scan only (D-03); POST /webhooks/square is unaffected |
| T-06-07 | Spoofing | X-Forwarded-For header manipulation to bypass IP rate limit | accept | Single Railway instance behind Railway's own proxy; header is set by Railway, not the client. Bypass risk is low at this scale; Redis-backed limiter is deferred |
| T-06-08 | Elevation of Privilege | Production ADMIN_PASSWORD leaked via Railway logs | mitigate | Env vars are never logged in the codebase; Railway dashboard access is the control plane — protect Railway account with MFA |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | express-rate-limit install gated behind human-verify checkpoint in Plan 06-01 Task 1; [ASSUMED] per slopcheck unavailability |
